# Transport-node telemetry — wire spec for client apps

This document describes how a `reticulum-lora-transport` node publishes
network-health telemetry over the air, so a client app (web, mobile,
desktop) can subscribe to it and render a fleet view.

The schema below is **frozen** — fields can be added at the tail in
later versions, but existing indices and meanings are stable. Treat
extra fields as forward-compatible noise and ignore them.

---

## 1. What a node emits

Each transport node owns one telemetry "destination" and emits two
distinct streams of Reticulum announces against it.

### 1.1 Cadence

| Stream | Period | `app_data` | Purpose |
|---|---|---|---|
| Alive  | 5 min  | empty      | Keeps relays' `path_table` warm to this destination (RNS spec §7.5) so the next full beacon propagates instantly. |
| Beacon | 2 h    | msgpack telemetry payload (see §4) | Carries the metrics. |

A client interested in metrics can ignore the alive stream. A client
that just wants "is this node still up?" can use either — the alive
stream is the lower-latency liveness signal.

### 1.2 Source

The beacon is generated by `src/rns/Telemetry.cpp` and scheduled in
`src/main.cpp` via `Transport::schedule_announce`. The encoder is
the only place that defines the wire format; if this doc and the
encoder disagree, the encoder wins.

## 2. Identifying a telemetry announce

A telemetry announce is a normal Reticulum announce packet (HEADER_1,
BROADCAST, SINGLE, ANNOUNCE, RNS spec §2 + §4) addressed to a
**deterministic destination_hash** derived from the node's identity.

### 2.1 destination_hash derivation

Per RNS spec §1.2:

```
app_name        = "transport.telemetry"               # ASCII, no NUL
name_hash       = SHA256(app_name)[:10]               # 10 bytes
identity_hash   = SHA256(node_public_key)[:16]        # 16 bytes
destination_hash = SHA256(name_hash || identity_hash)[:16]
```

`node_public_key` is the 64-byte concatenation `X25519_pub(32) ||
Ed25519_pub(32)` carried in the announce body. Once a client has
seen one announce from a node it can compute the telemetry
`destination_hash` for that node and subscribe.

A client building a fleet view typically:

1. Listens for announces on any destination.
2. Filters announces whose `name_hash` (first 10 bytes after the
   public key in the body) equals `SHA256("transport.telemetry")[:10]`.
3. Records `(destination_hash → public_key)` mapping for later
   address-bar / pretty-name use.

### 2.2 Pre-computed `name_hash`

```
SHA256("transport.telemetry")[:10] = c3 4e 7a c5 2e a4 e2 f7 96 3f
```

(Recompute in your client's own SHA-256 to avoid depending on this
constant.)

## 3. Extracting `app_data` from an announce

Per RNS spec §4.1, announce body layout is:

```
public_key(64) | name_hash(10) | random_hash(10) | [ratchet_pub(32)] | signature(64) | app_data(*)
```

`ratchet_pub` is present only when the announce header's
context_flag bit (0x20) is set. `app_data` runs from the end of the
signature to the end of the packet body. Length is implicit — read
to EOF.

For an **alive** announce, `app_data` is zero bytes.
For a **beacon** announce, `app_data` is the msgpack blob defined in
§4.

You should verify the signature (Ed25519 over `dest_hash ||
public_key || name_hash || random_hash || [ratchet_pub] || app_data`,
spec §4.2) before trusting any field. Unsigned beacons MUST be
discarded.

## 4. Telemetry payload (msgpack)

`app_data` of a beacon is a single msgpack **5-element fixarray**
with explicit-width numeric tags. The encoder always emits these
exact tags — no smallest-fits compression — so byte layout is
deterministic.

```
[0] lat              : float32  OR  nil
[1] lon              : float32  OR  nil
[2] battery_mv       : uint16
[3] route_count      : uint16
[4] packets_forwarded: uint32
```

### 4.1 Wire byte layout

| Byte(s) | Hex | Meaning |
|---|---|---|
| 0 | `95` | fixarray of length 5 |
| 1 | `ca` *or* `c0` | float32 tag (lat) **or** nil |
| 2..5 | 4 bytes BE | lat IEEE-754 float32 (omitted if nil) |
| next | `ca` *or* `c0` | float32 tag (lon) **or** nil |
| +4 | 4 bytes BE | lon IEEE-754 float32 (omitted if nil) |
| next | `cd` | uint16 tag (battery_mv) |
| +2 | 2 bytes BE | battery millivolts |
| next | `cd` | uint16 tag (route_count) |
| +2 | 2 bytes BE | route count |
| next | `ce` | uint32 tag (packets_forwarded) |
| +4 | 4 bytes BE | packets forwarded |

Position-only (no msgpack-rpc map keys), so element order is the
schema. Do not reorder.

### 4.2 Worked example

A node at lat 37.7749° / lon -122.4194° with 3.95 V battery,
12 known routes, 4231 packets forwarded:

```
95
ca 42 17 18 b4         # float32  37.7749
ca c2 f4 d6 e9         # float32 -122.4194
cd 0f 6c               # uint16  3948 mV
cd 00 0c               # uint16  12
ce 00 00 10 87         # uint32  4231
```

Total: 23 bytes.

A node with no position fix configured:

```
95
c0                     # nil
c0                     # nil
cd 0f 6c               # uint16  3948 mV
cd 00 0c               # uint16  12
ce 00 00 10 87         # uint32  4231
```

Total: 15 bytes.

## 5. Field semantics

### `lat` / `lon`
- Units: decimal degrees, IEEE-754 float32.
- Range: lat ±90, lon ±180.
- `nil` means the operator hasn't set a position via the webapp.
  Render the node as "unlocated" — do **not** map it to (0, 0).
- Source: manually configured via the webapp. There is no GPS in
  the firmware — these fields will not change without a config commit.

### `battery_mv`
- Units: millivolts at the battery rail (post-divider, post-ADC
  calibration).
- Range: 0..65535. Practical range for Li-ion: ~2800–4200.
- 0 means the board has no battery sense (e.g. running off USB on a
  variant with `HAS_BATTERY_SENSE == 0`). Treat as "unknown".

### `route_count`
- Units: count.
- Definition: number of entries in the node's `PathTable` at the
  moment the beacon fires (RNS spec §6 path table — destination
  → next-hop mapping).
- Range: 0..65535. A healthy mesh node typically reports a few to a
  few hundred.

### `packets_forwarded`
- Units: count, cumulative since last boot.
- Definition: sum of these `Transport::Stats` counters at emit time:
  - `announces_rebroadcast`
  - `data_forwarded_header_1`
  - `data_forwarded_header_2`
  - `proof_forwarded`
  - `link_data_forwarded`
- Resets to 0 on every reboot (counters live in RAM, never persisted
  per the flash-wear policy).
- A client tracking deltas should detect reboots by watching for
  counter regression: if `packets_forwarded[t+1] < packets_forwarded[t]`
  for the same `destination_hash`, the node restarted.

## 6. Decoder snippets

### 6.1 JavaScript (msgpack-lite)

```js
import msgpack from "msgpack-lite";

function decodeTelemetry(appDataBytes) {
  if (appDataBytes.length === 0) return { kind: "alive" };

  const arr = msgpack.decode(appDataBytes);
  if (!Array.isArray(arr) || arr.length < 5) {
    throw new Error("telemetry: not a 5+ element array");
  }
  const [lat, lon, battery_mv, route_count, packets_forwarded] = arr;
  return {
    kind: "beacon",
    position: lat == null || lon == null ? null : { lat, lon },
    battery_mv,
    route_count,
    packets_forwarded,
  };
}
```

### 6.2 Python (msgpack)

```python
import msgpack

def decode_telemetry(app_data: bytes):
    if not app_data:
        return {"kind": "alive"}
    arr = msgpack.unpackb(app_data, raw=False)
    if not isinstance(arr, list) or len(arr) < 5:
        raise ValueError("telemetry: not a 5+ element array")
    lat, lon, batt, routes, fwd = arr[:5]
    return {
        "kind": "beacon",
        "position": None if lat is None or lon is None else (lat, lon),
        "battery_mv": batt,
        "route_count": routes,
        "packets_forwarded": fwd,
    }
```

### 6.3 Kotlin (org.msgpack:msgpack-core)

```kotlin
import org.msgpack.core.MessagePack

data class Telemetry(
    val position: Pair<Float, Float>?,
    val batteryMv: Int,
    val routeCount: Int,
    val packetsForwarded: Long,
)

fun decodeTelemetry(appData: ByteArray): Telemetry? {
    if (appData.isEmpty()) return null            // alive announce
    MessagePack.newDefaultUnpacker(appData).use { u ->
        val arity = u.unpackArrayHeader()
        require(arity >= 5) { "telemetry: not a 5+ element array" }

        val lat = if (u.nextFormat.valueType.isNilType) { u.unpackNil(); null } else u.unpackFloat()
        val lon = if (u.nextFormat.valueType.isNilType) { u.unpackNil(); null } else u.unpackFloat()
        val batt = u.unpackInt()
        val routes = u.unpackInt()
        val fwd = u.unpackLong()

        val pos = if (lat != null && lon != null) Pair(lat, lon) else null
        return Telemetry(pos, batt, routes, fwd)
    }
}
```

## 7. Versioning and forward compatibility

If the schema gains fields, they will be appended to the array (e.g.
index `[5]`, `[6]`, ...). Decoders MUST tolerate longer arrays —
extract the first five fields and ignore the rest. Decoders MUST NOT
hard-fail when the array length exceeds 5.

If a future change is incompatible, it ships under a new app_name
(e.g. `transport.telemetry.v2`) so old and new beacons coexist on
the air, and old clients keep working.
